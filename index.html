klog helps you to keep track of your projects

klog is designed to be a districuted issue tracking and time management tool. as such, it should integrate with any distributed version control system. I use git, but klog is vcs agnostic - just track the files along with your code!

Installation

currently klog is available as a perl script, which you can download and either package with your project (to ensure all members can use it), or install on your system.

we are planning to port it to nodejs soon, so it will be available via npm.

Basic usage

klog is designed to make your issue tracking easier by having a simple command line interface

we are planning to implement a web interface soon!

Issue tracking

files are stored as plain text files, witha filename based on the exact time the issue was raised (millisecond precision) and the firstbits of a hash generated from the authors email address. these firstbits become the UID of the issue, preventing colision with other issues being raised by other developers at the same time.

by keeping all the issues with the git repo, it becomes a great offline tool for managing a project. any developer who has the repo, also has the tools needed to track progress, pick a task to work on, and read the rationale behind chages to the code. this feature is also great for archiving projects, with the confidence that all the related discussion and material is all in one place!

Time management (work log)

one issue that keeps coming up for me personally, is how to effectively track work activity, and time spent on a project. after trying out several solutions, from online log files, spreadsheets, and time tracking software, I ended up reverting to plain old time-stamped log files. after programming a hotkey to insert the date I would just write down when I started working, then each thing I did as a brief description line, and when I finished. this would constitute a `block` of work, and I would then calculate the time spent on each block. this system has been working great for me, but I could not help feeling it could be much better.

my current best idea, is to try to link the process of tracking issues, making commits, and managing time, as they are all very much integral to the work process. when I am working on a bug, I want that work to be related to a `block` that can be identified how long, and when it took place, and also the code and comments relating to the work. filling the gap here is what klog is all about.

this project is very much in it's early stages and should be considered a development project, not yet fit for production.
